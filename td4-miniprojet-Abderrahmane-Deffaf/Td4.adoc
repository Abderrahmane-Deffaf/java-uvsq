== Description du Mini Moteur Algébrique

=== Les opérateurs

Le MMA évaluera trois types d’opérateurs :

* **PROJECT** : prend une table et une liste d’attributs *LA*. Il produit en sortie une table contenant uniquement la liste d’attributs *LA*.

* **SELECT** : permet une sélection. Il prend une table et un prédicat simple constitué d’un attribut comparé à une valeur constante.
  Il n’y a pas d’opérateurs de conjonction, de disjonction ni de négation.
  Trois comparateurs sont disponibles : `=`, `<` et `>` pour les nombres, et `=` pour les chaînes.
  La valeur constante peut être un entier ou une chaîne. Toute chaîne est entourée de guillemets simples (`'`).

* **JOIN** : prend deux tables et deux attributs à joindre. Seule l’équi-jointure est implémentée.
  Il produit une table résultant de la jointure des deux tables.

Si besoin, vous pouvez vous référer à http://sql.bdpedia.fr/alg.html[algèbre relationnelle].


=== Contraintes sur les Répertoires de Requêtes

Votre application, livrée sous la forme d’un fichier JAR avec ses dépendances, sera exécutée depuis un répertoire contenant les tables requises par la ou les requêtes, ainsi que les fichiers de requêtes.

[source,bash]
----
$ tree test4
test4
├── orders.csv
├── reqSelUsers.json
├── results
│   └── output.csv
└── users.csv
----

Dans l'exemple ci-dessus:
 * Le fichier `reqSelUsers.json` contient une requête dont le format sera décrit ci-dessous. 
 * Le répertoire `results` contiendra un fichier https://fr.wikipedia.org/wiki/Comma-separated_values[CSV] nommé `output` (respectez bien la typographie, sinon votre note sera impactée, car l'automate d'évaluation de votre projet ne pourra pas retrouver les données requises). 
 * Les fichiers `orders.csv` et `users.csv` sont des fichiers de données utilisés par les requêtes. 

Exemple de contenu d'un fichier csv:
[source,bash]
----
$ cat users.csv 
id,name,age
1,Alice,22
2,Bob,30
3,Charlie,27
4,David,19
----

Nous supposerons que les fichiers CSV sont toujours de petite taille et peuvent tenir en mémoire.

=== Une Requête Algébrique

Toute requête est décrite dans un fichier https://en.wikipedia.org/wiki/JSON[JSON
].
Dans un répertoire qui évalue un plan algébrique, il peut y avoir une ou plusieurs requêtes.
Une requête est exécutée dans le répertoire contenant l’ensemble des éléments nécessaires.
La requête est exécutée à l’aide de la commande `java``.

Par exemple :

[source,bash]
----
monPC@td4_alg/test3
$ java -jar ../target/td4_alg-1.0-SNAPSHOT-jar-with-dependencies.jar reqProjSel.json 
Résultat écrit dans results/output.csv
----

Dans l'exemple ci-dessus, la machine Java s’exécute depuis le répertoire `test3` un "jar" fichier JAR et ses dépendances sur la requête `reqProjSel.json`

[source,bash,title="reqProjSel.json"]
----
{
  "tables": {
    "users": "users.csv",
    "orders": "orders.csv"
  },
  "plan": {
  "type": "PROJECT",
  "columns": ["name"],
  "input": {
    "type": "SELECT",
    "predicate": "age < 25",
    "input": {
      "type": "SCAN",
      "table": "users"
    }
  }
}
----

Une requête est définie par un fichier `JSON` comportant, dans une première partie, la désignation des fichiers CSV contenant les données, et dans une seconde partie, la requête algébrique.  
L'exemple ci-dessus illustre une requête qui effectue une sélection, suivie d'une projection sur le nom de la collection `users`.

[source,bash,title="optJoin.json"]
----
{
  "tables": {
    "users": "users.csv",
    "orders": "orders.csv"
  },
  "plan": {
    "type": "PROJECT",
    "columns": ["name", "amount"],
    "input": {
      "type": "JOIN",
      "left": {
        "type": "PROJECT",
        "columns": ["id", "name"],
        "input": {
          "type": "SCAN",
          "table": "users"
        }
      },
      "right": {
        "type": "SCAN",
        "table": "orders"
      },
      "onLeft": "id",
      "onRight": "user_id"
    }
  }
}
----

L’exemple de la requête `optJoin.json` ci-dessus effectue une projection depuis la table users. Le résultat est ensuite joint avec la table orders, produisant le résultat final.

== Une interface pour Exprimer des Requêtes

Cette seconde partie vise à traduire les requêtes algébriques, écrites en impératif, en un fichier de requête algébrique au format JSON.

Si l'utilisateur qui utilise son editeur de texte, exprimer une requête comme par exemple:
[source,bash,title="reqProjSel.alg"]
----
users: users.csv
orders: orders.csv

R1 = SELECT users age < 25
R = PROJ R1 name
----

Lors du lancement de la commande:
[source,bash]
----
ty:~/mkTDs/td4_alg/test3
$ java -jar ../target/td4_alg-1.0-SNAPSHOT-jar-with-dependencies.jar reqProjSel.alg 
Traduction en reqProjSel.json
----


* Votre application doit détecter l’extension `.alg` et effectuer la traduction. Elle n’exécute pas la requête, celle-ci étant dévolue à la première partie du projet.
* Chaque argument est séparé par un espace. Ainsi, on exprime jamais "age<25"
* Une chaine de caractère est mise entre des quotes ''. Il ne peut être qu'en dernière position.


=== Autre exemple de traduction
La requête `optJoin.json` equivaut à la traduction suivante :

[source,bash]
----
$ cat optJoin.alg 
users: users.csv
orders: orders.csv

R1 = PROJ users id name
R2 = JOIN R1 orders id user_id
R = PROJ R2 name amount
----

== Evaluation de votre Projet

* Un automate lancera la construction de votre projet, puis appliquera plusieurs requêtes algébriques. Si le résultat de l’évaluation est conforme, vous obtiendrez des points. Dans le cas contraire, aucune réclamation ne sera possible. Pour rappel:
  .  Votre code doit fonctionner suivant les exigences décrites dans "Contraintes sur les Répertoires de Requêtes". L'automate lancera votre jar dans un répertoire contenant une requête et effectura une évaluation.

  . Vos fichiers de données CSV en entrée ainsi que le fichier CSV de sortie doivent être "propres". Cela signifie qu'il ne doit pas y avoir d'espace avant, en fin de chaîne, au début et à la fin d'une valeur d'attribut (qui sont séparée **obligatoirement par une virgule**). Par exemple : "albert, 16, rue du poisson" n'est pas correct. Il faut: "albert,16,rue du poisson". Pas de quote autour des chaines de caractères.

  . L'automate s'attend, pour les tests JUnit 5, à voir une ligne du style `Tests run: 1, Failures: 0, Errors: 0, Skipped: 0`. Si votre pom.xml est mal configuré et qu'il ne voit pas cette ligne, la notation en tiendra compte.

* L'automate peut également vérifier:
   - les tests unitaires ainsi que votre mise en œuvre, en se basant sur l'approche fonctionnelle.

   - si votre code respecte les normes de style et les exigences de couverture des tests.

   - si vous avez effectué des commits git régulièrement

   - la similitude (plagiat) des codes produits par vous tous. Le but ce cet effort est d'apprendre de nouvelles notions qui ne peuvent être acquises en copiant.

== À compléter ici:

=== Quelles bibliothèques ont été utilisées et pourquoi ?

J'ai utilisé **Gson** pour lire et écrire les fichiers JSON car c'est simple et efficace. **JUnit 5** pour tester mon code, **Checkstyle** pour respecter les standards de codage, et **JaCoCo** pour voir quelle partie du code est testée. Le plugin **Maven Shade** me permet de créer un seul JAR avec toutes les dépendances.

=== Décrire les relations d'héritage ou autres entre vos classes

Tous mes opérateurs (SCAN, SELECT, PROJECT, JOIN) implémentent l'interface `Operator` qui définit une méthode `execute()`. Chaque opérateur retourne un objet `Table`. J'ai utilisé la composition : les opérateurs s'enchaînent les uns dans les autres pour former un arbre d'exécution.

=== Quel est le rôle des différentes classes ?

**Opérateurs** : `ScanOperator` lit les CSV, `SelectOperator` filtre les lignes, `ProjectOperator` sélectionne les colonnes, `JoinOperator` fait les jointures.

**Modèle** : `Table` représente une table de données.

**CSV** : `CsvReader` et `CsvWriter` lisent et écrivent les fichiers CSV.

**Exécution** : `ExecuteQuery` lit le JSON et exécute la requête.

**Conversion** : `BuildJsonFromAlg` convertit les fichiers .alg en JSON.

**Point d'entrée** : `App` détecte le type de fichier et lance le bon traitement.

=== Pourquoi Json est bien/pas bien adapté pour l'interprétation un plan d'execution ?

JSON est bien adapté car il représente naturellement la structure en arbre des opérateurs imbriqués. C'est lisible, facile à parser avec Gson, et flexible pour ajouter de nouveaux opérateurs. Le seul inconvénient c'est qu'il peut devenir verbeux pour des requêtes très complexes, mais pour notre projet ça convient parfaitement.

=== Quelles classes avez vous utilisées pour la lecture d'un Json ?

J'utilise `Gson` pour parser le fichier initial, puis `JsonObject` pour naviguer dans la structure du plan, et `JsonArray` pour lire les listes de colonnes. C'est simple et ça marche bien.

=== Comment consulter le rapport de couverture de code par les tests ?

Après `mvn test`, j'ouvre le fichier `target/site/jacoco/index.html` dans mon navigateur. Il me montre quelles lignes de code sont testées et lesquelles ne le sont pas, avec des pourcentages par classe et par package.

=== Quels avantages apporte une approche de programmation non impérative, mais fonctionnelle, pour l'évaluation des requêtes dans votre projet ?

L'approche fonctionnelle rend le code plus clair : chaque opérateur prend des données et produit un résultat sans modifier l'original. J'utilise beaucoup les streams (`.filter()`, `.map()`) qui sont concis et expressifs. Les prédicats comme `Predicate<Map<String, String>>` me permettent de passer des conditions en paramètre. Et surtout, c'est beaucoup plus facile à tester car il n'y a pas d'effets de bord.

=== Y a-t-il des bibliothèques Java qui permettraient de lire un fichier CSV et de l'instancier comme une collection ?

Oui, il y a Apache Commons CSV, OpenCSV, Jackson CSV, etc. Mais j'ai préféré coder mon propre `CsvReader` car nos CSV sont simples, ça m'a permis d'apprendre les I/O en Java, et je n'avais pas besoin de fonctionnalités avancées.

=== Comment avez-vous géré la variabilité du nombre de colonnes dans une table représentée dans une collection ?

J'utilise `List<String>` pour les noms de colonnes et `List<Map<String, String>>` pour les lignes. Chaque ligne est un `LinkedHashMap` qui garde l'ordre des colonnes et s'adapte automatiquement au nombre de colonnes. Comme ça, une table peut avoir 2 ou 20 colonnes sans problème.

=== Quelles évolutions peut-on envisager ?

On pourrait ajouter d'autres opérateurs comme GROUP BY, ORDER BY, ou DISTINCT. Supporter des conditions plus complexes avec AND/OR. Optimiser les requêtes en réorganisant automatiquement les opérateurs. Gérer de vrais types de données au lieu de tout traiter en String. Et pourquoi pas une interface graphique pour construire les requêtes visuellement.
